一、项目定位与目标

本项目围绕伦敦道路交通安全场景，构建一个可复现、可解释、可在线演示的事故严重度分类系统，重点回答事故在既定上下文条件下更可能呈现何种严重程度。

系统目标包含三层。第一层是形成稳定的数据治理链路并保证数据可审计。第二层是建立多模型对比基线并输出一致的评估证据。第三层是将方法、结果和边界转化为可读的研究叙事，支持展示与沟通。

二、数据来源与数据构成

主数据来自英国交通部公开发布的 STATS19 道路安全数据集，覆盖 collision、vehicle、casualty 三类记录，时间段采用 2020 至 2024 年已验证年度数据，空间范围以伦敦相关记录为主。

天气数据用于补充事故发生时段附近的环境条件，节假日数据来自 GOV.UK Bank Holidays JSON 接口并用于识别假期效应，时间衍生特征由事故原始日期和时间字段直接生成。

数据模块 关键字段 作用
Collision accident_index, accident_severity, date, time, 经纬度, 道路信息 提供目标标签与事故上下文
Vehicle vehicle_type, manoeuvre, junction_location, age_of_driver 补充车辆与驾驶行为线索
Casualty casualty_class, casualty_type, casualty_severity, age_of_casualty 补充伤亡结构信息并支持一致性检查
Weather + Holiday temperature, precipitation, radiation, is_bank_holiday 补充环境与日历因素

三、方法设计与技术路线

  1. 数据准备阶段将三张主表按业务键和时间字段对齐后形成统一主表，同时记录每一步 join 的匹配率与字段完整率。
  2. 数据治理阶段统一处理异常编码与缺失结构并保留治理台账，从而保证目标标签合法、来源可追溯、样本口径一致。
  3. 特征工程阶段在原始字段基础上构建时间上下文、天气条件和道路环境相关特征，并形成可复用的训练输入模板。
  4. 建模阶段使用 Logistic Regression、Random Forest、Histogram Gradient Boosting 三种方法建立可比较基线并保持接口一致。
  5. 验证阶段同时采用分层交叉验证与时间外推切分，以便分别评估统计稳健性与真实部署场景下的泛化能力。
  6. 解释阶段通过混淆矩阵、误差样本表、特征重要性和阈值分析共同还原模型行为并支撑结果解读。

四、检验方式与质量控制

项目检验采用数据检验、训练检验、产物一致性检验三条主线并行推进。数据检验会固定输出标签合法率、异常值计数、join 匹配率和关键字段缺失率。训练检验会固定输出 Macro F1、Balanced Accuracy、致命伤害类（Fatal）Precision、Recall、F1 以及校准相关指标。产物一致性检验会核对文档结论与 artifact 文件数值一致，避免叙事与证据脱节。

为避免结果偏差，流程中加入了泄漏检查报告，并区分可在事故前获得的特征与事故后才可观察到的字段，从而保证主要模型的输入边界清晰且应用语义一致。

五、结果解释与业务意义

当前系统已经形成可稳定复现的基线结果，并可在同一界面完成数据质量查看、模型比较、误差追踪与阈值讨论。该结果可支持两类核心任务，其一是快速识别不同条件下的严重度风险分布，其二是帮助说明哪些因素在样本中更常与高严重度事件共同出现。

通过将模型输出与误差样本并列展示，系统能够把“模型给出什么结论”与“结论来自哪些证据”对应起来，这一证据链结构可直接用于研究汇报、岗位展示和后续迭代决策。

六、使用流程与展示建议

本项目可按数据构建、模型训练、结果评估、文档展示四步运行，推荐先运行数据与训练脚本生成 artifacts，再基于说明书中的图表与表格完成结论核验，最后按问题背景、数据构成、质量控制、方法比较、误差分析、边界与迭代方向的顺序进行展示。

在讲解过程中可围绕问题定义、数据处理、方法选择、证据解释、可复现性五个维度展开，使每个结论都能落在具体图表或文件输出上。

八、项目边界与迭代方向

系统边界定义为伦敦范围内的公开可得数据驱动分类任务，目标是提供结构化风险识别与研究证据组织能力。后续迭代优先方向包括提升天气数据匹配质量、强化时间外推表现、优化致命伤害类（Fatal）召回与概率校准，并继续扩展可解释特征与数据版本管理。

1. Project Positioning and Goal

This project builds a reproducible, interpretable, and web-deployable accident severity classifier for the London road safety context, with a clear objective to estimate which severity level is more likely under observed conditions.

The objective has three layers. The first layer is a fully traceable data governance pipeline. The second layer is a consistent multi-model baseline with comparable evaluation evidence. The third layer is a research narrative that connects method, result, and boundary in a readable way.

2. Data Source and Data Structure

The core data comes from the UK Department for Transport STATS19 open data, including collision, vehicle, and casualty records, using validated annual data from 2020 to 2024 with London-focused filtering.

Weather variables provide environmental context near accident time, bank holiday signals come from the GOV.UK JSON endpoint, and additional temporal features are derived directly from event date and time fields.

Module Key Fields Role
Collision accident_index, accident_severity, date, time, coordinates, road attributes Target label and event context
Vehicle vehicle_type, manoeuvre, junction_location, age_of_driver Vehicle and driving behavior signals
Casualty casualty_class, casualty_type, casualty_severity, age_of_casualty Injury structure and consistency checks
Weather + Holiday temperature, precipitation, radiation, is_bank_holiday Environmental and calendar context

3. Method Design and Technical Route

  1. Data preparation aligns the three core tables with business keys and time fields to produce one master table while recording join match rates and field completeness.
  2. Data governance standardizes invalid codes and missing patterns with a persistent audit log so that label legality, provenance, and sample scope remain consistent.
  3. Feature engineering combines native columns with temporal context, weather conditions, and road environment signals to create a reusable training feature template.
  4. Modeling uses Logistic Regression, Random Forest, and Histogram Gradient Boosting under a unified interface to establish a comparable baseline suite.
  5. Validation applies stratified cross-validation and time-based holdout together to quantify both statistical stability and deployment-like generalization.
  6. Interpretation links confusion matrices, error-case tables, feature importance, and threshold analysis to explain model behavior with direct evidence.

4. Validation and Quality Control

The project uses a three-track quality framework across data checks, training checks, and artifact consistency checks. Data checks report label validity, invalid-code counts, join match rates, and missingness of key fields. Training checks report Macro F1, Balanced Accuracy, Fatal class Precision, Recall, F1, and calibration metrics. Artifact consistency checks ensure that narrative statements are numerically aligned with artifact files.

A leakage check is integrated into the pipeline, and pre-event features are separated from post-event fields so that model inputs preserve clear semantic boundaries for practical use.

5. Result Interpretation and Practical Value

The current system provides a stable baseline that can be reproduced and inspected in one place, including data quality evidence, model comparison, error tracing, and threshold behavior. This supports two practical tasks, rapid understanding of severity risk patterns under different conditions and transparent explanation of factors that co-occur with higher severity outcomes.

By presenting model outputs together with error examples, the system builds a direct link between each conclusion and its evidence file, which improves clarity for research reporting and professional presentation.

6. Runbook and Presentation Flow

The recommended sequence runs in four stages, master table build, model training, model evaluation, and document presentation. A practical flow then follows problem framing, data structure, quality control, method comparison, error analysis, and iteration planning.

For communication, structure the narrative with problem framing, data process, method choice, evidence reading, and reproducibility so that every statement points to a specific figure or artifact.

8. Project Boundary and Iteration Path

The project boundary is a London-focused severity classification pipeline based on publicly available data, with a primary function to provide structured risk identification and evidence-ready research outputs. The next iteration path prioritizes weather join quality, stronger time-holdout performance, better Fatal class recall and calibration, and richer interpretable features with versioned data governance.
